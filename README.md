# Архітектура модулів Node.js — Практичний челендж

Цей репозиторій містить два повністю незалежних Fastify-сервери. Деяка функціональність спрощена, нас цікавить робота з модулями. Один побудований на **CommonJS (CJS)**, інший - на **ECMAScript Modules (ESM)**.

Обидва сервери мають ідентичну архітектуру (Routes → Controllers → Services → Repositories). Однак наразі вони **не запускаються**.

Ці помилки створені, щоб побачити відмінності між CJS та ESM.
Вам потрібно запустити сервер на CJS і ESM.
---

## Як практикуватися

1.  `cd cjs` або `cd esm`.
2. Виконайте `npm install`.
3. Запустіть сервер: `npm run dev`.
4. Прочитайте вивід помилки, знайдіть баг та виправте його!
5. Повторюйте, поки сервер не запуститься і ендпоінти `/api/users` та `/api/health` не працюватимуть.

---

## Охоплені концепції

Це завдання перевіряє ваше розуміння відмінностей між CJS та ESM:

### 1. Форма експорту/імпорту модулів
* **CJS:** Перезапис `exports = {}` руйнує внутрішню прив'язку до `module.exports`.
* **ESM:** Змішування іменованих (`export const`) та дефолтних (`export default`) експортів і неправильний їх імпорт.
* **ESM:** Спроба використати дефолтний імпорт для функції, яка була експортована як іменований експорт.

### 2. Резолюція директорій та шляхів
* **CJS:** Пропуск відносного префікса (як `../`) змушує Node шукати в `node_modules`.
* **ESM:** Відсутність явного розширення `.js` у локальному імпорті викликає `ERR_MODULE_NOT_FOUND`.
* **ESM:** Імпорт папки (наприклад, `./plugins/error-handler`) падає, бо ESM не резолвить автоматично до `index.js`, як CJS.

### 3. Виконання файлів та кешування стану
* **CJS:** `require` кешує модулі. Мутація експортованого об'єкта конфігурації в одному файлі назавжди змінює стан сервера всюди.
* **ESM:** ESM використовує **Live Bindings**, а не статичні копії. Спроба перезаписати імпортований примітив (`count++`) викликає `TypeError: Assignment to constant variable`.

### 4. Циклічні залежності
* **CJS:** Коли два модулі синхронно викликають `require()` один одного, Node перериває нескінченний цикл, повертаючи порожній, неповний об'єкт `{}`. Це викликає помилки типів при виклику методів пізніше.
* **ESM:** Коли два модулі циклічно імпортують один одного, і один намагається використати інший під час top-level, Node кидає **Temporal Dead Zone (TDZ)** Reference Error.

### 5. Особливості нативного середовища
* **CJS та ESM:** Змінні `--env-file` у Node 20+  - це строго рядки. Слабко типізовані фолбеки (`process.env.PORT || '8080'`) можуть спричинити тихі невідповідності типів у фреймворках як Fastify.

### 6. Інтероперабельність Node 24+ / Динамічний require
* **CJS:** Ви *можете* синхронно викликати `require()` для ESM-файлу в сучасному Node. Але якщо цей ESM-файл використовує **Top-Level Await**, він миттєво падає з `ERR_REQUIRE_ASYNC_MODULE`. Потрібно використовувати динамічний асинхронний `await import()`.
* **ESM:** `__dirname` та `__filename` не існують в ESM. Їх потрібно відтворити з `import.meta.url`.

### 7. Імпорт JSON
* **CJS:** `require('./data.json')` неявно парсить JSON.
* **ESM:** `import data from './data.json'` падає з `ERR_IMPORT_ASSERTION_TYPE_MISSING`. Потрібно використовувати `with { type: "json" }`.
